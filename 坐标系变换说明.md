## 1. 三个坐标系

系统中同时存在三个坐标系，所有坐标转换都在这三者之间进行。

### 1.1 GPS 地理坐标系

- 表示方式：`(latitude, longitude, altitude)`，单位度/度/米
- 用途：无人机真实地理位置、地图中心配置
- 特点：全球唯一，但不适合做向量运算

### 1.2 NED 坐标系（PX4 使用）

- 全称：North-East-Down
- 轴向：
  - X 轴 → 正北方向（North）
  - Y 轴 → 正东方向（East）
  - Z 轴 → 向下为正（Down）
- 单位：米
- 原点：**PX4 的 `vehicle_odometry` 话题以飞机上电时的位置为原点**（这是问题的根源，见第 3 节）
- 用途：PX4 内部飞控、ROS2 话题数据

### 1.3 UE5 世界坐标系

- 轴向：
  - X 轴 → 前方（Forward，对应地图北方）
  - Y 轴 → 右方（Right，对应地图东方）
  - Z 轴 → 向上为正（Up）
- 单位：**厘米**
- 原点：UE5 世界原点 `(0, 0, 0)` 必须对应地图中心 GPS 坐标 `(lat0, lon0, alt0)`
- 用途：UE5 场景中所有 Actor 的位置

## 2. 坐标转换公式

### 2.1 NED ↔ UE5 世界坐标

这是最简单的转换，只涉及轴向映射和单位换算。

**NED → UE5（用于遥测显示，A 链路）**

```
UE5_X (cm) =  NED_N (m) × 100     // 北 → 前
UE5_Y (cm) =  NED_E (m) × 100     // 东 → 右
UE5_Z (cm) = -NED_D (m) × 100     // 下 → 上，取负！
```

**UE5 → NED（用于控制指令，B 链路）**

```
NED_N (m) =  UE5_X (cm) × 0.01
NED_E (m) =  UE5_Y (cm) × 0.01
NED_D (m) = -UE5_Z (cm) × 0.01    // 上 → 下，取负！
```

> 关键点：Z 轴方向相反，转换时必须取负。

### 2.2 四元数姿态转换（NED → UE5）

PX4 的四元数表示从 FRD（前右下）车身坐标系到 NED 参考系的旋转，格式为 `[w, x, y, z]`。

UE5 的 `FQuat` 构造函数参数顺序为 `(x, y, z, w)`，且 UE5 是左手坐标系。

转换步骤：

```cpp
// 1. 重排顺序：PX4 [w,x,y,z] → UE5 FQuat(x,y,z,w)
FQuat Q(QuatData[1], QuatData[2], QuatData[3], QuatData[0]);

// 2. 翻转 Z 分量（NED Down → UE5 Up，坐标系手性变化）
FQuat UE5Quat(Q.X, Q.Y, -Q.Z, Q.W);

// 3. 转欧拉角后翻转 Yaw（右手系 → 左手系）
FRotator Euler = UE5Quat.Rotator();
Euler.Yaw = -Euler.Yaw;
```

### 2.3 GPS → NED（相对地图中心）

这是整个坐标系统的核心，用于计算飞机上电点相对于地图中心的偏移。

**输入**：飞机 GPS `(lat, lon, alt)`，地图中心 GPS `(lat0, lon0, alt0)`

**第一步：GPS → ENU（东北天）**

```python
import math

dlat = lat - lat0
dlon = lon - lon0

lat0_rad = math.radians(lat0)
meters_per_deg_lat = 111319.0                          # 纬度每度约 111319 米
meters_per_deg_lon = 111319.0 * math.cos(lat0_rad)    # 经度每度随纬度变化

E = dlon * meters_per_deg_lon   # 东方向偏移（米）
N = dlat * meters_per_deg_lat   # 北方向偏移（米）
U = alt - alt0                  # 天方向偏移（米）
```

**第二步：ENU → NED**

```python
NED_N = N
NED_E = E
NED_D = -U    # 天 → 下，取负
```

**精度说明**：上述公式是简化的平面近似（忽略地球曲率），在 10km 范围内误差 < 1m，满足无人机演示场景需求。如需更高精度，可使用 `pyproj` 库的 `Transformer`。

## 3. 问题根源：为什么 odometry 不能直接用

PX4 的 `vehicle_odometry` 话题中，`position` 字段的含义是：

> **飞机当前位置相对于上电时位置的 NED 偏移**

也就是说，每架飞机上电时 `position = (0, 0, 0)`，之后的数值是相对于上电点的位移。

**后果**：

```
飞机A 在操场西北角上电 → odometry.position = (0, 0, 0)
飞机B 在操场东南角上电 → odometry.position = (0, 0, 0)

两架飞机都静止不动，但 UE5 里如果直接用 odometry 坐标，
它们会显示在同一个位置！
```

**解决方案**：引入地图中心 `(lat0, lon0, alt0)` 作为全局原点，利用 GPS 数据计算每架飞机上电点相对于地图中心的偏移，然后叠加到 odometry 上：

```
corrected_position = raw_odometry_position + gps_offset
```

其中 `gps_offset` = 飞机上电点相对于地图中心的 NED 坐标（由第 2.3 节公式计算）。

## 4. 完整数据流

### 4.1 A 链路（遥测显示：PX4 → UE5）

```
PX4 vehicle_odometry.position (NED, 以上电点为原点)
    ↓
[Python: drone_data_bridge.py]
    订阅 vehicle_global_position 获取飞机 GPS (lat, lon, alt)
    计算 gps_offset = GPS_to_NED(lat, lon, alt, lat0, lon0, alt0)
    corrected_position = raw_position + gps_offset
    将 corrected_position 写入 YAML 的 position 字段
    ↓ UDP YAML
[UE5: RealTimeDroneReceiver]
    调用 DroneCoordinateService->NedToWorld(corrected_position)
    = FVector(N*100, E*100, -D*100)
    设置 Actor 世界坐标
```

### 4.2 B 链路（控制指令：UE5 → PX4）

```
[UE5: DroneOpsPlayerController]
    用户点击地图，获取 UE5 世界坐标 (X, Y, Z) cm
    调用 DroneCoordinateService->WorldToNed(WorldPos)
    = FVector(X/100, Y/100, -Z/100) m
    打包为 32 字节 UDP 发送到 8899
    ↓
[Python: multi_ue_controller.py → ue_to_px4_bridge.py]
    接收 NED 目标坐标
    减去 gps_offset（还原为以上电点为原点的相对坐标）
    发布 TrajectorySetpoint 给 PX4
```

> 注意 B 链路中 `ue_to_px4_bridge.py` 需要减去 `gps_offset`，因为 PX4 的 TrajectorySetpoint 也是以上电点为原点的。这部分 `ue_to_px4_bridge.py` 中的 `GPSOffsetManager` 已经处理，逻辑是 `apply_to_position` 加偏移，控制端需要反向减偏移。

## 6. `DroneCoordinateService` 接口说明

```cpp
// 设置地图中心（由 BP_MapBootstrap 在 BeginPlay 时调用）
void SetMapCenter(double Lat, double Lon, double Alt);

// 检查是否已初始化
bool IsReady() const;

// NED (米) → UE5 世界坐标 (厘米)
// 输入：以地图中心为原点的 NED 坐标
// 输出：UE5 世界坐标，原点 (0,0,0) 对应地图中心
FVector NedToWorld(FVector NedMeters) const;
// 实现：return FVector(NedMeters.X * 100, NedMeters.Y * 100, -NedMeters.Z * 100);

// UE5 世界坐标 (厘米) → NED (米)
FVector WorldToNed(FVector WorldCm) const;
// 实现：return FVector(WorldCm.X * 0.01, WorldCm.Y * 0.01, -WorldCm.Z * 0.01);

// GPS → UE5 世界坐标 (厘米)
// 内部先做 GPS→NED，再做 NED→World
FVector GpsToWorld(double Lat, double Lon, double Alt) const;

// UE5 世界坐标 → GPS（近似，用于信息面板显示）
FGeographicCoordinates WorldToGps(FVector WorldCm) const;

// 获取当前地图中心配置
FMapCenterConfig GetMapCenter() const;
```

## 7. 配置一致性要求

UE5 端和 Python 端必须使用**完全相同**的地图中心坐标，否则两端的坐标系会错位。

| 配置位置 | 字段 |
|----------|------|
| UE5：`DA_MapRuntimeConfig` | `MapCenterLatitude` / `MapCenterLongitude` / `MapCenterAltitude` |
| Python：`Config/multi_drone_config.yaml` | `map_center.latitude` / `map_center.longitude` / `map_center.altitude` |

`EUW_MapCenterConfigurator` 的"导出"按钮负责将 UE5 端配置同步写入 YAML，确保两端一致。

## 8. 常见错误与排查

| 现象 | 原因 | 排查方法 |
|------|------|----------|
| 多架飞机在 UE5 中叠在一起 | `drone_data_bridge.py` 未应用 GPS 偏移 | 检查 YAML 中 `gps_offset_applied` 字段是否为 true |
| 飞机位置与地图严重偏移 | UE5 和 Python 的地图中心不一致 | 对比 `DA_MapRuntimeConfig` 和 `multi_drone_config.yaml` 中的坐标 |
| 飞机位置正确但控制目标点偏移 | B 链路未正确减去 GPS 偏移 | 检查 `ue_to_px4_bridge.py` 的 `GPSOffsetManager` 是否收到 GPS 数据 |
| `DroneCoordinateService->IsReady()` 返回 false | `BP_MapBootstrap` 未在场景中放置，或 `DA_MapRuntimeConfig` 未配置 | 检查关卡中是否有 `BP_MapBootstrap`，检查资产路径 |
| 飞机姿态方向错误 | 四元数转换中 Z 分量或 Yaw 未取负 | 检查 `QuatToEuler` 函数，对照第 2.2 节 |
