# 多机地图交互与阵列控制架构改造清单

## 1. 文档目的

本文档用于把《多机地图交互与阵列控制需求文档》中的 5 个重点需求，拆解为可实施的架构改造清单。

本文档重点回答 4 个问题：

- 需要新建哪些 C++ 类、蓝图、Widget、数据资产和输入资产
- 需要改动哪些现有类、配置和脚本
- 每个新增/改动项的输入、输出、核心数据结构是什么
- 5 个需求之间哪些能力应共享，避免后续重复造轮子

---

## 2. 当前项目现状

当前工程已经具备以下基础：

- 已启用 `CesiumForUnreal` 插件
- 已有 `AUE5DroneControlCharacter` 负责本地玩家无人机控制和 UDP 发送
- 已有 `ARealTimeDroneReceiver` 负责 YAML 遥测接收、NED 到 UE 坐标转换和模型驱动
- 已有 `AMultiDroneManager`、`AMultiDronePlayerController`、`AMultiDroneCharacter` 作为多机原型
- 已有 `Variant_Strategy` 样板，可复用其 RTS 式相机、选择和 HUD 思路

### 2.1 当前通讯架构（旧版）

旧版架构采用”UE5直接向各桥接端口发送”的方式：
```
UE5 → ue_to_px4_bridge.py ×3 → PX4 ×3
     (分别发送到8889/8891/8893)
```

### 2.2 新多机通讯架构

新架构引入主控制器，实现统一发送和掩码分发：
```
UE5 仿真环境 (发送选择掩码 + 目标位置)
  ↓ UDP 32字节 (端口 8899)
multi_ue_controller.py (主控制器，运行在Windows)
  ↓ 解析位掩码，分别转发到各桥接端口
  • 8889 → px4_1 (掩码bit 0)
  • 8891 → px4_2 (掩码bit 1)
  • 8893 → px4_3 (掩码bit 2)
ue_to_px4_bridge.py ×3 (3个实例，分别对应3架飞机)
  ↓ ROS2 话题 (各自的话题前缀)
PX4 真实无人机 ×3 (px4_1, px4_2, px4_3)
```

**新架构核心特性**：
- UE5统一发送到8899端口，简化多机控制逻辑
- 32字节协议包含：timestamp(double) + x/y/z(float) + mode(uint32) + drone_mask(uint32) + sequence(uint32)
- 主控制器解析掩码，自动分发到对应桥接端口
- 支持单机控制（掩码1位为1）和多机控制（掩码多位为1）
- 桥接器继续使用24字节协议与PX4通讯

当前主要问题：

- 运行架构仍以 `TopDown` 模板为核心，业务状态没有独立的多机层
- 相机、输入、控制、通信、选中状态、UI 状态分散在多个 Actor 内
- 多机能力目前主要停留在”切换镜头/路由发送者”，还没有统一注册表、统一状态模型和统一 UI
- 坐标转换逻辑主要放在接收器中，尚未沉淀为全局服务
- 阵列、信息面板、地图状态、自由镜头等需求还没有共享底座
- 需要适配新的32字节掩码协议和主控制器架构
- **坐标系统对齐问题**：
  - UE5端没有明确的地图中心配置机制
  - Python桥接端通过命令行参数 `--map-center-lat/lon` 设置地图中心
  - 缺少UE5与Python配置一致性验证
  - 当前 `RealTimeDroneReceiver` 使用相对偏移，但没有统一的坐标基准服务
  - 多机场景下，各无人机的坐标参考系可能不一致

结论：

- 不建议继续把 5 个 FR 直接叠加在 `BP_TopDownCharacter` / `BP_TopDownController` 上
- 建议新增一层 `DroneOps` 业务架构，作为正式运行入口
- 必须适配新的多机通讯架构，实现基于掩码的统一控制
- **必须建立统一的坐标系统**：
  - 在UE5端实现 `DroneCoordinateService` 全局服务
  - 通过 `DA_MapRuntimeConfig` 配置地图中心GPS坐标
  - 确保UE5世界原点 `(0,0,0)` 对应地图中心 `(lat0, lon0, alt0)`
  - 提供配置导出/验证工具，保证UE5与Python配置一致

## 3. 目标分层架构

建议按以下 6 层组织：

1. 地图层

- 负责 Cesium 场景初始化、地图加载状态、地图坐标锚点和世界对齐

2. 无人机注册与状态层

- 负责维护 `DroneId -> Sender / Receiver / Telemetry / Selection / LockState`

3. 输入与交互层

- 负责点击无人机、点击地图、中键信息面板、F 键自由镜头、多选和阵列播放控制

4. 相机层

- 负责“跟随镜头”和“自由镜头”的切换、回退和过渡

5. 执行与调度层

- 负责单机目标点下发、阵列序列播放、控制抢占和暂停/停止

6. 展示层

- 负责地图加载状态、当前控制对象、无人机信息面板、镜头模式和阵列状态显示

## 5. 新建资产总表

### 5.1 共享 C++ 资产

### `DroneOpsTypes.h`

- 类型：共享枚举与结构体头文件
- 用途：承载 5 个 FR 的统一数据模型
- 输入：控制请求、遥测快照、阵列配置、镜头状态
- 输出：供 Controller、Subsystem、Widget、数据资产统一引用
- 核心数据结构：
- `EDroneCameraMode`：`Follow` / `Free`
- `EDroneAvailability`：`Online` / `Offline` / `Lost`
- `EDroneControlLockReason`：`None` / `FormationPlayback` / `Offline` / `InvalidTarget`
- `FDroneDescriptor`：`Name`、`DroneId`（整数ID）、`MavlinkSystemId`、`BitIndex`、`ThemeColor`、`UEReceivePort`、`ControlPort`、`TopicPrefix`
- `FDroneTelemetrySnapshot`：`DroneId`、`Availability`、`WorldLocation`、`NedLocation`、`GeographicLocation`（GPS）、`Velocity`、`Attitude`、`Altitude`、`LastUpdateTime`
- `FDroneTargetCommand`：`DroneId`、`TargetWorldLocation`、`TargetNedLocation`、`TargetGeographicLocation`（可选）、`Mode`、`IssuedAt`
- `FMultiDroneControlPacket`：`Timestamp`（double）、`X`、`Y`、`Z`（float）、`Mode`（uint32）、`DroneMask`（uint32）、`Sequence`（uint32）【32字节UDP包结构，与README一致】
- `FMapCenterConfig`：`Latitude`（double）、`Longitude`（double）、`Altitude`（double）、`bIsInitialized`【地图中心配置】
- `FCameraModeState`：`CameraMode`、`FollowDroneId`、`LastFollowLocation`、`LastFollowRotation`
- `FFormationMemberSlot`：`DroneId`、`RoleName`、`RelativeOffset`
- `FFormationStep`：`StepId`、`DurationSec`、`MemberTargets`
- `FFormationExecutionState`：`PresetId`、`StepIndex`、`ElapsedSec`、`bPaused`、`ParticipantIds`

### `DroneRegistrySubsystem.h/.cpp`

- 类型：`UGameInstanceSubsystem`
- 用途：全局无人机注册表和状态中心
- 输入：无人机注册、遥测更新、选中切换、控制锁定、相机跟随目标变更
- 输出：查询接口和广播事件
- 对外接口建议：
- `RegisterSenderPawn(DroneId, PawnRef)`
- `RegisterReceiverActor(DroneId, ReceiverRef)`
- `UpdateTelemetry(DroneId, Snapshot)`
- `SetPrimarySelectedDrone(DroneId)`
- `SetMultiSelectedDrones(DroneIds)`
- `ApplyControlLock(DroneId, LockReason)`
- `GetPrimarySelectedDrone()`
- `GetTelemetry(DroneId)`
- `GetAllDroneDescriptors()`
- `GetSelectedDroneMask()` - 根据当前选中的无人机生成32位掩码
- `GetDroneBitIndex(DroneId)` - 获取无人机在掩码中的位索引
- `GetDroneByBitIndex(BitIndex)` - 根据位索引获取DroneId
- 广播事件建议：
- `OnDroneRegistered`
- `OnPrimarySelectionChanged`
- `OnTelemetryUpdated`
- `OnControlLockChanged`

### `DroneCoordinateService.h/.cpp`

- 类型：坐标转换服务类
- 用途：统一处理 `Cesium 地理坐标`、`UE 世界坐标`、`PX4 NED 坐标`
- 输入：地图点击位置、无人机遥测、阵列偏移、地图锚点配置
- 输出：供 FR-01、FR-02、FR-05 复用的标准转换结果
- 核心数据：
- `MapCenterGPS`：地图中心GPS坐标（lat0, lon0, alt0）
- `WorldOrigin`：UE5世界原点对应的地理坐标
- `NorthAlignment`：北向对齐规则（UE5 X轴对应NED哪个方向）
- `AltitudeReference`：高度基准（海拔或相对地面）
- `CesiumGeoreference`：Cesium地理参考组件引用
- 对外接口建议：
- `SetMapCenter(double Lat, double Lon, double Alt)` - 设置地图中心GPS坐标
- `GetMapCenter()` - 获取当前地图中心
- `WorldToNed(FVector WorldLocation)` - UE5世界坐标 → NED坐标（米）
- `NedToWorld(FVector NedLocation)` - NED坐标 → UE5世界坐标（厘米）
- `WorldToGeographic(FVector WorldLocation)` - UE5世界坐标 → GPS坐标
- `GeographicToWorld(double Lat, double Lon, double Alt)` - GPS坐标 → UE5世界坐标
- `IsCoordinateSystemReady()` - 检查坐标系统是否已初始化
- 架构说明：
- 必须与Python桥接端的 `--map-center-lat/lon` 参数保持一致
- 建议从 `DA_MapRuntimeConfig` 读取地图中心配置
- 初始化时应验证Cesium地理参考是否正确设置
- 所有坐标转换必须通过此服务，避免分散在各Actor中

### `DroneSelectionComponent.h/.cpp`

- 类型：无人机交互组件
- 用途：承载单选/多选/高亮状态
- 输入：点击命中、拖框命中、清空选择、阵列编辑选择
- 输出：高亮状态、UI 可读的选择状态
- 核心数据：
- `DroneId`
- `bIsPrimarySelected`
- `bIsSecondarySelected`
- `SelectionThemeColor`

### `DroneTelemetryComponent.h/.cpp`

- 类型：遥测状态组件
- 用途：把 `ARealTimeDroneReceiver` 解析出的实时数据抽象成可绑定状态
- 输入：YAML 遥测解析结果
- 输出：Widget、状态栏、信息面板可直接读取的快照
- 核心数据：
- `CurrentSnapshot`
- `LastOnlineTime`
- `bHasValidTelemetry`

### `DroneCommandSenderComponent.h/.cpp`

- 类型：控制发送组件
- 用途：构建多机控制包并统一发送到主控制器
- 输入：`FDroneTargetCommand` 或多机选择掩码 + 目标位置
- 输出：32字节UDP控制包到端口8899（multi_ue_controller.py）
- 核心数据：
- `UnifiedControllerIP`：主控制器IP地址（默认127.0.0.1）
- `UnifiedControllerPort`：主控制器端口（固定8899）
- `LastIssuedCommand`
- `SendInterval`
- 对外接口建议：
- `SendTargetCommand(Command)` - 单机模式，根据DroneId构建掩码
- `SendMultiTargetCommand(SelectionMask, TargetLocation, Mode)` - 多机模式，直接使用掩码
- `SendHoverCommand(DroneId)` - 悬停命令
- 架构说明：
- 不再为每架无人机维护独立的RemoteIP/RemotePort
- 统一发送到8899端口，由multi_ue_controller.py负责解析掩码并分发
- 单机控制时，通过DroneRegistrySubsystem查询BitIndex并构建掩码（仅该位为1）
- 多机控制（阵列）时，直接传入包含多个无人机的掩码

### `DroneFreeCameraPawn.h/.cpp`

- 类型：自由镜头 Pawn
- 用途：FR-04 的正式镜头载体
- 输入：`W/A/S/D`、`Q/E`、鼠标 Look、初始位置/旋转
- 输出：相机变换、镜头状态
- 核心数据：
- `MoveSpeed`
- `Acceleration`
- `LookSensitivity`
- `CurrentVelocity`
- 关键要求：
- `W/A/S/D` 必须相对于当前视角方向移动
- `Q/E` 控制垂直升降

### `DroneOpsPlayerController.h/.cpp`

- 类型：正式业务控制器
- 用途：统一管理 FR-02、FR-03、FR-04、FR-05 输入与状态机
- 输入：点击、悬停、中键、F 键、阵列播放控制、自由镜头移动
- 输出：选中状态变更、目标点派发、信息面板打开、自由镜头切换、阵列调度
- 核心数据：
- `PrimarySelectedDroneId`
- `MultiSelectedDroneIds`
- `HoveredDroneId`
- `CameraModeState`
- `FreeCamRef`
- `TargetMarkerRef`
- 不建议继续直接堆在现有 `AMultiDronePlayerController` 上

### `DroneOpsGameMode.h/.cpp`

- 类型：正式业务 GameMode
- 用途：切换到新的 DroneOps 运行入口
- 输入：默认 Pawn、PlayerController、HUD、地图初始化配置
- 输出：统一业务框架类

### `FormationPresetDataAsset.h/.cpp`

- 类型：`UPrimaryDataAsset`
- 用途：存储内置阵列预设和自定义序列结构
- 输入：编辑器配置、运行时载入

- 核心功能实现：
  - **点击地图模式**：
    - 启用后，在Viewport中点击Cesium地图
    - 通过 `ACesiumGeoreference::TransformUeToLongitudeLatitudeHeight()` 获取GPS坐标
    - 自动填充到输入框
  - **自动配置**：
    - 创建或加载 `DA_MapRuntimeConfig` 数据资产
    - 设置 `MapCenterLatitude/Longitude/Altitude`
    - 标记资产为已修改（`MarkPackageDirty()`）
    - 保存资产
  - **Cesium同步**：
    - 查找场景中的 `ACesiumGeoreference` Actor
    - 设置 `OriginLatitude/Longitude/Height` 为地图中心
    - 触发Cesium重新加载地图瓦片
  - **配置导出**：
    - 生成YAML格式配置文件
    - 保存到 `Config/multi_drone_config.yaml`
    - 包含 `map_center` 节点
  - **可视化调试**：
    - 在Viewport中绘制地图中心标记（红色球体）
    - 绘制北向箭头（蓝色，指向+X方向）

### `multi_ue_controller.py`

- 类型：新增主控制器脚本
- 职责：接收UE5统一控制包，解析选择掩码，分发到各桥接端口
- 运行位置：Windows本地（与UE5同机）
- 输入：
- 32字节UDP包，端口8899
- 结构：`Timestamp`（double）+ `X/Y/Z`（float×3）+ `Mode`（uint32）+ `DroneMask`（uint32）+ `Sequence`（uint32）
- 输出：
- 根据掩码位，将目标点转换为24字节包分别发送到：
  - 端口8889 → px4_1（掩码bit 0）
  - 端口8891 → px4_2（掩码bit 1）
  - 端口8893 → px4_3（掩码bit 2）
- 核心逻辑：
- 解析32字节包，提取DroneMask和目标位置
- 遍历掩码的每一位，若为1则向对应桥接端口发送24字节控制包
- 24字节包格式：`struct '<dfffi'`（Timestamp + X + Y + Z + Mode）
- 配置来源：
- 从 `multi_drone_config.yaml` 读取各无人机的 `bit_index` 和 `control_port`
- 架构说明：
- 这是新架构的核心分发层，实现"一对多"控制
- UE5侧只需维护一个发送端口（8899），简化了多机控制逻辑
- 支持单机控制（掩码仅1位为1）和多机控制（掩码多位为1）

#### T1-01 数据层：`DA_MapRuntimeConfig` + `DroneCoordinateService`

**负责人**：C++ 工程师

**目标**：建立全局坐标系统的唯一数据源和转换服务，是整个 FR-01 的核心底座。

#### T1-02 Python 端：`drone_data_bridge.py` 加入 GPS 偏移

**负责人**：Python 工程师

**目标**：让 A 链路（遥测显示）发给 UE5 的 YAML 数据，携带经过地图中心校正的绝对 NED 坐标，而不是以飞机上电点为原点的相对坐标。

**背景**：当前 `drone_data_bridge.py` 直接转发原始 `vehicle_odometry`，其 `position` 字段以飞机上电点为原点。`ue_to_px4_bridge.py` 里已有完整的 `GPSOffsetManager` 实现，可直接复用。

**需要改动**：`drone_data_bridge.py`
1. 新增订阅 `{topic_prefix}/fmu/out/vehicle_global_position`（与 `ue_to_px4_bridge.py` 相同方式）
2. 复用或内联 `GPSOffsetManager`（从 `ue_to_px4_bridge.py` 提取）
3. 在转发 YAML 前，对 `position` 字段调用 `gps_manager.apply_to_position(raw_position)`
4. 从 `multi_drone_config.yaml` 的 `map_center` 节点读取地图中心，与 `ue_to_px4_bridge.py` 保持一致
5. YAML 中新增可选字段 `gps_offset_applied: true`，便于 UE5 端判断数据来源

**输入**：`vehicle_global_position` 话题（lat/lon/alt）、`map_center` 配置

**输出**：发往 UE5 的 YAML `position` 字段变为相对地图中心的绝对 NED 坐标（米）

#### T1-03 UE5 接收端：`RealTimeDroneReceiver` 改造

**负责人**：C++ 工程师（依赖 T1-01 完成）

**目标**：将 `RealTimeDroneReceiver` 的坐标计算从”以第一帧 NED 为原点的相对偏移”改为”以地图中心为原点的绝对坐标”，通过 `DroneCoordinateService` 完成转换。

**当前问题**（见 `RealTimeDroneReceiver.cpp:549-593`）：
- `ReferencePosition` = 第一帧 NED，后续只算相对偏移
- `InitialLocation` = Actor 在编辑器中的摆放位置
- 最终位置 = `InitialLocation + NEDToUE5(CurrentNED - ReferencePosition)`
- 这导致无人机在 UE5 中的绝对位置与真实 GPS 位置无关

**需要改动**：`RealTimeDroneReceiver.cpp`
1. 在 `BeginPlay` 中获取 `DroneCoordinateService` 引用，检查 `IsReady()`
2. 删除 `ReferencePosition` 和 `InitialLocation` 的相对偏移逻辑
3. `ProcessPacket` 中改为：`NewTarget = CoordService->NedToWorld(DroneData.Position)`
4. 保留 `NEDToUE5()` 方法但标记为 deprecated，内部改为调用 `DroneCoordinateService`
5. 若 `DroneCoordinateService` 未就绪，回退到旧的相对偏移逻辑并打印警告

**输入**：YAML 中已校正的绝对 NED 坐标（由 T1-02 保证）

**输出**：`BP_RealTimeDrone` 在 UE5 世界中的位置与真实 GPS 位置对齐

**验收**：多架飞机上电后静止，UE5 中各飞机的相对距离与实际摆放距离一致（误差 < 1m）

### 需要改动的现有资产

- `DroneOpsPlayerController`：编辑模式下拦截地图点击，改为添加航点而非派发目标点

### 核心数据结构

- `FWaypoint`：`WorldLocation(FVector)` / `DwellTimeSec(float)` / `bOverrideAltitude(bool)`
- `FDronePath`：`DroneId(int32)` / `Waypoints(TArray<FWaypoint>)` / `CruiseSpeedMps(float)`
- `FPathSequence`：`SequenceId(FString)` / `DisplayName(FString)` / `DronePaths(TArray<FDronePath>)` / `bIsPreset(bool)`
- `EFormationPlaybackState`：`Idle` / `Playing` / `Dwelling` / `Paused` / `Completed` / `Stopped`

### 架构说明

- 路径编辑器是**运行时 UI**，不是编辑器工具，在 PIE 和打包版本中均可使用
- 所有航点坐标存储为 **UE5 世界坐标（厘米）**，由 `DroneCoordinateService` 保证与地图对齐
- 播放引擎在 UE5 侧做调度，Python 侧继续扮演”单机目标点桥接器”，不感知路径概念
- 播放时每个航点单独发送一次控制命令，到达后再发下一个，不做轨迹插值（首版）
- 抢占规则（明确定义）：路径播放中的无人机处于锁定状态，FR-02 单机派发被拒绝；`StopPlayback()` 解锁，无人机切换悬停
- 编辑模式与正常操作模式互斥：打开路径编辑器时，地图点击行为切换为”添加航点”，关闭后恢复”派发目标点”

## 9. 最终建议

本项目最关键的不是“多写几个蓝图”，而是先把共享业务层建立起来。

如果继续沿用当前做法，把点击、相机、目标点、遥测、信息面板、阵列都堆在 `TopDownCharacter`、`PlayerController` 和 `RealTimeDroneReceiver` 中，后续 5 个需求会互相耦合，维护成本会快速上升。

推荐路线：

- 新建 `DroneOps` 正式业务层
- 保留 `TopDown` 和当前 `MultiDrone*` 作为兼容或原型层
- 用“注册表 + 坐标服务 + 控制发送组件 + 遥测组件 + 正式 HUD”作为长期底座
